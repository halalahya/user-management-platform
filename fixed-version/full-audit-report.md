## 1. 项目概述

### 1.1 项目简介

本项目是一个基于 Python Flask 框架和 SQLite3 数据库开发的简易用户信息管理系统。系统经历了多轮功能迭代，依次实现了用户登录、注册、搜索、头像上传、个人中心、充值、动态页面加载等核心功能。在每次功能开发过程中，同步进行安全审计与漏洞修复，累计发现并修复了 **31 项安全漏洞**。

本报告对所有已发现漏洞进行系统性汇总，涵盖漏洞原理、攻击方式、风险等级、修复方案和前后对比，为 Web 安全课程实训提供完整参考。

### 1.2 运行环境

|:---|:---|
| 开发语言 | Python 3.10+ |

## 3. 基础安全漏洞

### 3.1 密码明文存储与硬编码凭据

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-BASE-01 |
| **风险等级** | ⛔ 高危 |
| **漏洞原理** | 用户密码以明文形式硬编码在 USERS 字典中，登录时通过 == 直接比对 |
| **原始代码** | `"password": "admin123"` → `if USERS[username]["password"] == password` |
| **修复方式** | 使用 `generate_password_hash()` 加盐哈希存储，`check_password_hash()` 安全比对 |
| **修复代码** | `USERS["admin"]["password"] = generate_password_hash(admin_pw)` |

### 3.2 通信层未加密

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-BASE-02 |
| **风险等级** | ⛔ 高危 |
| **漏洞原理** | 未配置 SSL 证书，所有通信使用明文 HTTP 协议 |
| **原始代码** | `app.run(host="0.0.0.0", port=5000)` |
| **修复方式** | 生成自签名证书，配置 `ssl_context` 参数启用 HTTPS |
| **修复代码** | `app.run(ssl_context=(SSL_CERT, SSL_KEY))` |

### 3.3 Session 密钥硬编码

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-BASE-03 |
| **风险等级** | ⚠️ 高危 |
| **漏洞原理** | secret_key 固定为 `"dev-key-2025"`，可被猜测伪造 session cookie |
| **原始代码** | `app.secret_key = "dev-key-2025"` |
| **修复方式** | 从环境变量读取，不存在则使用 `secrets.token_hex(32)` 随机生成 |
| **修复代码** | `app.secret_key = os.environ.get("FLASK_SECRET_KEY", secrets.token_hex(32))` |

### 3.4 密码明文回显至前端

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-BASE-04 |
| **风险等级** | ⚠️ 高危 |
| **漏洞原理** | 登录成功后完整用户信息（含密码）传递到模板并渲染到页面 |
| **原始代码** | `user_info = USERS[username]` → `{{ user.password }}` |
| **修复方式** | 新增 `sanitize_user_info()` 过滤 password 字段 |
| **修复代码** | 返回数据排除 password 和 email 字段 |

### 3.5 调试注释泄露凭据

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-BASE-05 |
| **风险等级** | ⚠️ 高危 |
| **漏洞原理** | login.html 页面 HTML 注释中直接写明管理员用户名密码 |
| **原始代码** | `<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->` |
| **修复方式** | 删除该 HTML 注释 |

### 3.6 缺少 CSRF 保护

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-BASE-06 |
| **风险等级** | ⚠️ 高危 |
| **漏洞原理** | 表单未包含 CSRF Token，可被跨站请求伪造攻击 |
| **原始代码** | `<form method="POST">` 无 CSRF 隐藏字段 |
| **修复方式** | `secrets.token_hex(32)` 生成 Token，全局注入模板，POST 时验证 |

### 3.7 Session Cookie 安全配置缺失

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-BASE-07 |
| **风险等级** | ⚠️ 高危 |
| **漏洞原理** | Cookie 未设置 HttpOnly、Secure、SameSite，会话无过期时间 |
| **原始代码** | 无相关配置 |
| **修复方式** | 四项配置一次性补全 |
| **修复代码** | `SESSION_COOKIE_HTTPONLY=True`、`SECURE=True`、`SAMESITE="Lax"`、`LIFETIME=2h` |

### 3.8 缺少点击挟持防护

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-BASE-08 |
| **风险等级** | ⚠️ 中危 |
| **漏洞原理** | 未设置 X-Frame-Options，页面可被嵌入 iframe |
| **修复方式** | 添加响应头 `X-Frame-Options: DENY` |

### 3.9 缺少 HSTS 头

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-BASE-09 |
| **风险等级** | ⚠️ 中危 |
| **漏洞原理** | 未设置 Strict-Transport-Security，浏览器可回退到 HTTP |
| **修复方式** | `Strict-Transport-Security: max-age=31536000; includeSubDomains` |

### 3.10 Content-Security-Policy 不完整

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-BASE-10 |
| **风险等级** | ⚠️ 中危 |
| **漏洞原理** | CSP 缺少 script-src、style-src 等核心指令 |
| **修复方式** | 设置包含 default-src、script-src、style-src 的完整策略 |

### 3.11 缺少 X-Content-Type-Options

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-BASE-11 |
| **风险等级** | 🔵 低危 |
| **漏洞原理** | 未设置 X-Content-Type-Options，浏览器可能执行 MIME 嗅探 |
| **修复方式** | `X-Content-Type-Options: nosniff` |

### 3.12 开发环境安全配置遗留

### 3.13 缺少表单输入校验

### 3.15 认证机制缺陷

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-BASE-15 |

| **漏洞原理** | 登录接口无失败次数限制，可暴力破解 |
| **修复方式** | 基于 IP 的登录失败计数，5 次失败后锁定 15 分钟 |

## 4. SQL 注入漏洞

### 4.1 UNION 注入

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-SQL-01 |
| **风险等级** | ⛔ 高危 |
| **漏洞位置** | `/search` 路由 |
| **漏洞原理** | keyword 参数通过 f-string 拼接，攻击者注入 `' UNION SELECT ...` 获取自定义数据 |
| **拼接后 SQL** | `SELECT ... WHERE username LIKE '%' UNION SELECT 1,'inj','inj@x.com','138'--%'` |
| **复现命令** | `curl "http://127.0.0.1:5000/search?keyword=%27%20UNION%20SELECT%201,%27inj%27,..."` |
| **修复方式** | 使用 `?` 占位符参数化查询 |

### 4.2 OR 万能注入

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-SQL-02 |
| **风险等级** | ⛔ 高危 |
| **漏洞位置** | `/search` 路由 |
| **漏洞原理** | 注入 `' OR '1'='1` 使 WHERE 条件恒真，返回全部用户数据 |
| **拼接后 SQL** | `SELECT ... WHERE username LIKE '%' OR '1'='1%' OR email LIKE '%' OR '1'='1%'` |
| **修复方式** | 使用 `?` 占位符参数化查询 |

### 4.3 注册页面注入

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-SQL-03 |
| **风险等级** | ⚠️ 中危 |
| **漏洞位置** | `/register` 路由 |
| **漏洞原理** | username 参数注入 `')--` 闭合 VALUES 列表，控制插入字段 |
| **拼接后 SQL** | `INSERT INTO users VALUES ('hacker', 'pass', 'h@x.com', '123')--', ...` |
| **修复方式** | 使用 `?` 占位符参数化查询 |

## 5. 文件上传漏洞

### 5.1 任意文件上传

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-FU-01 |
| **风险等级** | ⛔ 高危 |
| **漏洞原理** | 上传接口未校验文件扩展名，允许上传 .php/.py/.html 等任意类型文件 |
| **原始代码** | `filename = secure_filename(file.filename)` → 直接保存 |
| **复现方式** | `curl -F "file=@shell.php" http://127.0.0.1:5000/upload` |
| **修复方式** | 扩展名白名单 `ALLOWED_EXTENSIONS = {".jpg", ".jpeg", ".png", ".gif", ".webp"}` |

### 5.2 同名文件覆盖

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-FU-02 |
| **风险等级** | ⚠️ 中危 |
| **漏洞原理** | 直接使用用户提供的原始文件名，同名文件互相覆盖 |
| **修复方式** | 使用 `uuid.uuid4().hex` 生成唯一文件名 |

### 5.3 文件内容伪造

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-FU-03 |
| **风险等级** | ⚠️ 中危 |
| **漏洞原理** | 未验证文件是否真实图片，文本文件伪装 .png 可上传成功 |
| **修复方式** | 魔术头校验（Magic Bytes）：JPEG/PNG/GIF/WebP |

### 5.4 文件体积上限过大

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-FU-04 |
| **风险等级** | 🔵 低危 |
| **漏洞原理** | 16MB 上传上限过大，可被用于存储耗尽攻击 |
| **原始代码** | `app.config["MAX_CONTENT_LENGTH"] = 16 * 1024 * 1024` |
| **修复方式** | 下调至 5MB |

| 维度 | 内容 |

| **漏洞原理** | 导航栏"个人中心"链接硬编码 `user_id=1`（admin），普通用户点击跳转到 admin 页面 |
| **原始代码** | `<a href="/profile?user_id=1">个人中心</a>` |
| **修复方式** | 改为 `/profile` 无参链接，由路由自动解析当前登录用户 |

## 7. 动态页面加载漏洞

### 7.1 路径穿越读取任意文件

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-PG-01 |
| **风险等级** | ⛔ 高危 |
| **漏洞原理** | name 参数未过滤 `../`，可构造路径穿越读取服务器任意文件 |
| **原始代码** | `filepath = os.path.join("pages", name)` → 无校验直接读取 |
| **复现方式** | `curl "http://127.0.0.1:5000/page?name=../../etc/passwd"` |
| **修复方式** | 正则白名单 `^[a-zA-Z0-9_-]+$` + `os.path.realpath()` 路径前缀校验 |

### 7.2 敏感文件信息泄露

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-PG-02 |
| **风险等级** | ⛔ 高危 |
| **漏洞原理** | 读取到的文件内容未经脱敏直接返回给客户端 |
| **修复方式** | 配合 VULN-PG-01，路径限制在 pages/ 目录内 |

### 7.3 页面内容未限制渲染范围

| 维度 | 内容 |
|:---|:---|
| **漏洞编号** | VULN-PG-03 |
| **风险等级** | 🔵 低危 |
| **漏洞原理** | `{{ page_content \| safe }}` 禁用 HTML 转义 |

## 9. 安全总结与防护建议

### 9.1 安全开发规范总结

1. **密码必须哈希存储**，禁止任何形式的明文密码出现在源码中
2. **所有通讯必须走 HTTPS**，配合 HSTS 头锁定协议级别
3. **所有数据库操作必须使用参数化查询**，杜绝字符串拼接 SQL
4. **所有 POST/PUT 表单必须包含 CSRF Token**，前后端双重校验
5. **所有需要登录的接口必须校验 session**，防止未授权访问
6. **上传文件必须校验扩展名和内容**，白名单策略优于黑名单
7. **用户输入必须校验格式和范围**，特别是财务相关的数值参数
8. **文件路径拼接必须做白名单校验**，防止路径穿越
9. **Session Cookie 必须配置安全属性**：HttpOnly、Secure、SameSite
10. **调试注释和敏感信息不能在页面中暴露**

### 9.2 后续优化方向

| 优先级 | 优化项 | 说明 |
|:---|:---|:---|
| P0 | 正规 SSL 证书 | 使用 Let's Encrypt 等 CA 签发的受信任证书 |
| P0 | 数据库存储 | 用户数据迁移至 SQLite/MySQL，支持动态管理 |
| P1 | RBAC 权限管理 | 区分管理员与普通用户操作权限 |
| P1 | 登录日志审计 | 记录登录成功/失败日志 |
| P2 | 双因素认证 | 为管理员账户增加二次验证 |
| P2 | 密码策略强制 | 校验密码复杂度，强制定期更换 |
| P3 | Rate Limiting | 引入 Flask-Limiter 全局请求频率控制 |
| P3 | 文件杀毒扫描 | 上传文件落盘后调用 ClamAV 扫描 |
